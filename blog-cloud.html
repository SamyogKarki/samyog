Deploying a Full-Stack App Across AWS and Azure

July 22, 2024 Samyog Karki AWS, Azure, Linux, Full Stack

Why two clouds?

When I set out to deploy my full-stack web application, I wanted more than just a "works on my machine" demo. The goal was to understand, end-to-end, what it takes to run real software in production — configuring servers from scratch, tuning the database layer, and keeping the whole thing reliable under load. Running it on both AWS and Azure forced me to learn the primitives instead of clicking through a single provider's opinionated wizard.

Spinning up the Linux servers

On AWS I provisioned an EC2 instance; on Azure a comparable Linux VM. Both started as bare Ubuntu with nothing but SSH. Hardening came first — key-based auth only, UFW rules locked down to the ports I actually needed, and fail2ban watching for brute force. From there I installed the runtime, a reverse proxy, and the database, then wired up systemd units so everything would come back on reboot without me having to babysit it.

The small surprise: the two clouds feel more alike than different at this level. Networking primitives, disk types, security groups vs. network security groups — once you learn one, the other mostly maps. What differs is the paperwork around IAM and the cost model.

Making the database pull its weight

The app does the usual CRUD work, but the first version of my queries were lazy — no indexes on the columns I filtered on, and a few places where I was doing in-memory joins that belonged in SQL. Turning on query logging and watching what the app actually runs was the most useful thing I did. Adding three indexes cut the hot-path latency noticeably, and switching a couple of queries to parameterized prepared statements made the code both safer and faster.

High availability without the price tag

I didn't need a multi-region setup, but I did want the app to survive a single instance going sideways. Regular database backups dropped onto object storage, a tight restore script I actually tested, and a cron job checking both instances were answering. Simple, but enough to catch the kind of mistake that ruins an afternoon.

What I'd do differently

Next time I'd reach for Infrastructure as Code earlier — Terraform or even Ansible. Doing it by hand on two providers was a great learning exercise, but it also means the setup lives in my head. The whole point of going through all this was to build the muscle for running real systems, and the next step in that muscle is making the setup reproducible.
